Build a Firefox extension (Manifest V3) that protects users from IDN homograph attacks by blocking navigation to domains containing characters outside the scripts associated with the user's locale.
Core behaviour

Use browser.webRequest.onBeforeRequest with blocking (Firefox MV3 preserves this)
Intercept main_frame and sub_frame requests only
Extract the hostname, decode any punycode to Unicode, then check each character's Unicode Script property
Detect permitted scripts from navigator.languages (the full array, not just navigator.language)
Always permit Common and Inherited script categories (digits, hyphens, combining marks, etc.)
Always permit Latin regardless of locale, since the majority of legitimate domains use it
Block the request if the hostname contains any character whose script is not in the permitted set
On block, redirect to an internal extension page explaining what was blocked, showing the offending character(s) with their Unicode codepoint and script name

Locale → scripts mapping

Build a static mapping of locale codes to Unicode script sets (e.g. uk → [Cyrillic, Latin], ja → [Hiragana, Katakana, Han, Latin])
A user may have multiple preferred locales — union all their permitted script sets
The Common and Inherited scripts are always included regardless of locale

Whitelist

The block page must have a one-click "allow this domain" button
Whitelisted domains are stored in browser.storage.local in their Unicode (decoded) form
Matching at request time is done against the decoded hostname
Whitelisted entries are manageable (view/remove) from the extension's options page

Additional scripts UI

The options page allows the user to add extra permitted scripts beyond their locale default
Present this as a language picker (e.g. "Add Japanese") that silently maps to the correct Unicode script set underneath
This extends the same permitted scripts set used for blocking decisions

Architecture

manifest.json — MV3, permissions: webRequest, webRequestBlocking, <all_urls>, storage
background.js — request interception, script detection logic, whitelist lookup
blocked.html / blocked.js — block page with offending character detail and one-click whitelist button
options.html / options.js — whitelist management and additional language/script settings
unicode-scripts.js — the locale→scripts mapping table and character script detection logic, kept separate for testability

Start with unicode-scripts.js and the core detection logic, with a few test cases covering: a clean Latin domain, a mixed Cyrillic/Latin homograph attack domain, a legitimate Japanese domain, and a whitelisted domain that would otherwise be blocked. Get that working correctly before wiring up the webRequest plumbing.
